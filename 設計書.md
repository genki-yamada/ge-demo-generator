## 1. システム概要

**GE Demo Generator** は、自然言語で入力した業務課題から、**Gemini Enterprise** 向けの「動く AI エージェントデモ環境」一式をワンコマンドで生成・デプロイする低コードアクセラレーターである。

- **入力**: 顧客の業務課題（自然言語、多言語対応）
- **出力**: ドメイン特化の合成データ（BigQuery / Firestore）、MCP ツール群を備えた ADK エージェント、リアルタイム運用ダッシュボード（Data Viewer）を構築する単一の bash セットアップスクリプト
- **実行基盤**: 本体は Google Apps Script（GAS）上の Web アプリ。生成物はユーザー自身の Google Cloud プロジェクトに展開される

中核となる技術的特徴は **多層コード生成パイプライン**（GAS → bash → Python → LLM 指示）であり、本体は「コードを生成するコード」を生成する構造になっている。

## 2. 全体アーキテクチャ

システムは2つの独立した部分から成る。

```
┌────────────────────────────────────────────────────────┐
│ ① ジェネレーター本体（Google Apps Script Web アプリ）       │
│    - index.html : SPA（デモウィザード、MCPカタログ）        │
│    - Code.gs    : バックエンド（データ生成・スクリプト合成）  │
│         │                                                │
│         └─ generateSetupScript() ──────────┐            │
└─────────────────────────────────────────────┼───────────┘
                                              │ 生成
                                              ▼
        setup-demo-<domain>-<suffix>.sh（単一bashスクリプト）
                                              │ Cloud Shellで実行
                                              ▼
┌────────────────────────────────────────────────────────┐
│ ② 合成デモ環境（ユーザーの Google Cloud プロジェクト）      │
│    - BigQuery データセット + Firestore コレクション         │
│    - ADK エージェント（root/deep_analysis/background）     │
│    - FastAPI A2A サーバー（Cloud Run）                    │
│    - Data Viewer（Flask / Cloud Run Functions Gen2）     │
└────────────────────────────────────────────────────────┘
```

- **①ジェネレーター本体**: CE（カスタマーエンジニア）が操作する管理ツール。実リソースは一切作らず、設計とスクリプト生成のみ行う。
- **②合成デモ環境**: スクリプトが構築する実リソース。Gemini Enterprise からエージェントとして利用される。

**起動フロー（`doGet`, Code.gs:119）**

```

  → checkConfiguration()  // PROJECT_ID と LOG_SHEET_URL の存在確認
      ├─ 欠落あり → SetupError.html を返す
      └─ 正常    → index.html を返す（template 変数を注入）
```

`doGet` は `index` テンプレートに以下を注入する:

- `generatorModel`（CONFIG.MODEL、既定 `gemini-3.5-flash`）

`setXFrameOptionsMode(ALLOWALL)` を設定し、iframe 埋め込みを許可している。

|---|---|---|---|

### 3.3 デモ生成パイプライン

中心関数は `generateDemo(userGoal, options)`（Code.gs:487）。フロントから `google.script.run` で呼ばれる。

**オプション（`options`）**

| キー | 既定 | 説明 |
|---|---|---|
| `dataProfile` | `'standard'` | データ密度プロファイル（`getDataProfile_` で解決） |
| `rowCount` | プロファイル依存 | 生成行数（検証時は最大150行に制限） |
| `usePublicDataset` / `publicDatasetId` | `false` / `null` | 公開データセット連携 |
| `enableWorkspaceMcp` | `false` | Google Workspace MCP の有効化 |
| `importedMcpList` | `null` | インポート済み MCP サーバー一覧 |

**処理ステップ**

```
generateDemo()
 ├─ Step1: planAndGenerateData()       // Gemini でスキーマ・データ・指示文を計画生成
 ├─ Step2: validateGeneratedData()     // スキーマ整合検証・自動修復（最大150行）
 ├─ Step3: suffix/baseName/datasetId 採番
 │         suffix = UUID先頭8桁
 │         dirName   = "demo-<baseName>"
 │         datasetId = "demo_<baseName>"（ハイフン→アンダースコア）
 ├─ Step4: generateSetupScript()       // 全成果物を埋め込んだbashスクリプト合成
 ├─ classifyDemoTaxonomy_()            // 業種/ペルソナ/ユースケースを英語enumで分類
 └─ logUsageToSheet()                  // Usage_Logs へ記録
```

戻り値 `result` には `setupScript`、`dataPreview`、`rawTables`、`systemInstruction`、`businessInstruction`/`technicalInstruction`、`demoGuide`、`externalFiles`、`appliedFactors`、`firestore`、`metadata` 等が含まれる。

**関連サブ関数**
- `planAndGenerateData`（Code.gs:735）: 計画プロンプト（`buildPlanningPrompt`, Code.gs:1066）で Gemini を呼び、スキーマ・CSVデータ・システム指示文・デモガイド・外部ファイルを一括生成。
- `getTechnicalInstruction_`（Code.gs:846）: エージェントの A2UI/技術系システム指示文を生成（巨大な JS テンプレートリテラル）。
- `discoverPublicDataset`（Code.gs:640）/ `verifyAndResolveTable`（Code.gs:685）: Google 検索グラウンディングで実在の `bigquery-public-data` データセットを発見し、BigQuery API（Advanced Service）で実在検証。
- `validateGeneratedData`（Code.gs:1458）/ `validateAndRepairValue`（Code.gs:1610）: スキーマ型を踏まえた値の検証と自動修復。`parseCSVLine`、`repairTruncatedJson` など補助関数あり。

**分類体系（TAXONOMY, Code.gs:95）**: industry / persona / useCase の3軸を制御語彙（英語enum）で持つ。入力言語に関わらず英語値で記録され、index.html 側の `TAXONOMY` とミラーされる。

### 3.4 セットアップスクリプト合成（多層コード生成）

`generateSetupScript(params)`（Code.gs:1856〜15501、約13,600行）が本体の心臓部。単一の bash スクリプト文字列を組み立てて返す。

**多層エスケープ構造（最重要・AGENTS.md §2参照）**

```
Layer1: Code.gs（JS / GAS）      ─ テンプレートリテラル・文字列連結
   ↓
Layer2: bash スクリプト          ─ quoted/unquoted ヒアドキュメント
   ↓
Layer3: Python ソース            ─ ランタイム文字列操作
   ↓
Layer4: LLM システム指示文        ─ Gemini が消費
```

各層でバックスラッシュが倍化するため、`\n` の取り扱いが本コードベース最大のバグ源（`chr(10)` 使用が鉄則）。Data Viewer のみ Python の `"""` 層が1つ増えるため4バックスラッシュが必要。

## 5. 生成されるデモ環境（合成成果物）

### 5.1 データ層

```
BigQuery
 └─ Dataset: demo_<domain>_<suffix>
     ├─ Table 1..N（合成業務データ、US ロケーション）
     └─（任意）bigquery-public-data 参照

Firestore
 └─ Collection: demo-<domain>-<suffix>-data
     └─ 運用ドキュメント群（read-back / 書き込み対象）
```

### 5.2 エージェント（agent.py）

`__AGENT_EOF__` ヒアドキュメント（Code.gs:8060〜10398）で生成。三層エージェント構成。

| エージェント | モデル（env） | 役割 |
|---|---|---|
| `root_agent`（Coordinator） | `AGENT_MODEL_LITE`（既定 `gemini-3.5-flash`） | 挨拶・単純検索・概要回答・A2UI カード描画・ルーティング |
| `deep_analysis_agent`（Sub） | `AGENT_MODEL` | インライン専用の複雑多段推論。`register_background_task` 呼び出しは禁止（再帰エスカレーション防止） |
| `background_agent`（Worker） | `AGENT_MODEL` | スタンドアロン非同期ワーカー。サブエージェント・転送・A2UI 整形を持たない |

**ルーティング論理（指示文に符号化）**
- **概要/クイックルック**: root が1〜2クエリでインライン即答（deep へ転送しない）
- **深い多段分析**: 事前に Analysis Plan カードを描画 → ユーザー選択
  - 「Run Inline」→ `transfer_to_deep_analysis_agent`（root では分析 SQL を打たない）
  - 「Run in Background」→ 完全な `task_prompt` で `register_background_task`
- **SINGLE TASK RULE**: 1ワークフロー要求 = `register_background_task` 1回（`_norm_task_name` で重複ブロック）

**バックグラウンドパイプラインと Anti-Shallow Guard**
- ワークフロー: `SCAN → CLASSIFY → PROCESS → ESCALATE → NOTIFY → REPORT`（低リスクは自動実行、高リスクは人手承認へエスカレート）
- Anti-Shallow Guard（最終レポート前の自己チェック）: インラインは「3クエリ以上 / 2ソース以上参照 / サンドボックスで統計1回以上 / 具体数値で結論 / 3観点以上」、バックグラウンドは「5ツール呼び出し以上 / 3次元以上 / 統計2指標以上 / 定量インパクト付き推奨3件以上」を要求。未達なら深掘りへ戻る。

**コールバック / プラグイン**
- before_model: `_strip_part_metadata`（Vertex AI 互換のため part_metadata 除去）
- after_model: `inject_image_callback`（生成画像注入）、`a2ui_metadata_callback`（A2UI タグ検出）、`_enforce_task_result_text`（ツール結果のテキスト強制注入）
- before_tool: `_inline_tool_budget_gate`（インライン予算超過でツール遮断）、`_dedup_workspace_writes`（120秒重複抑止）
- after_tool: `_record_workspace_write`、`_log_bq_activity`（DML/Firestore 操作のアクティビティログ）
- before_agent（root のみ）: `_inject_completed_tasks`（未報告の完了タスクを次回応答冒頭で能動報告）
- plugins: `ReflectAndRetryToolPlugin()`、`LoggingPlugin()`
- `ContextCacheConfig(min_tokens=2048, ttl_seconds=3600, cache_intervals=20)`、`EventsCompactionConfig(compaction_interval=20, overlap_size=3)`

**A2UI スキーマ管理**: `A2uiSchemaManager(version=VERSION_0_8, catalogs=[BasicCatalog(examples_path="adk_agent/app/examples/0.8")])`。`generate_system_prompt(include_schema=True, include_examples=True, validate_examples=True)` でスキーマと例を指示文に注入。例 JSON は確認カード/分析サマリー/ダッシュボード/比較/フォーム/各種 Workspace 構成カードなど多数。

**コード実行**: `AgentEngineSandboxCodeExecutor`（pandas/numpy/scikit-learn/matplotlib。google.cloud 系は禁止）。

**カスタムツール（Python ローカル関数）**
- `generate_image(prompt)`: `gemini-3.1-flash-image`、16:9/JPEG。日本語プロンプト検出で画像内テキストを日本語強制。バイト列を session state に格納

- HTTP/2 無効化（`httpx.AsyncClient` を `http2=False`、timeout 300/connect 60）

### 5.4 A2A サーバー（fast_api_app.py）

`__FAST_API_EOF__`（Code.gs:10725〜約14979）で生成。

- **A2A サーバー**: `A2AFastAPIApplication`、Executor は `AdkAgentToA2AExecutor(A2aAgentExecutor)`。Agent Card は `/.well-known/agent.json`（`AgentCardBuilder`）。起動時に MCP 接続しない静的カード生成でハング回避。
- **A2UI ストリーム**: `A2uiStreamParser(catalog=...)` で逐次解析・JSON 修復・コンポーネント検証。失敗時は `<a2ui-json>...</a2ui-json>` 正規表現フォールバック。`application/json+a2ui` の DataPart に変換。
- **`/execute_task` バックグラウンドランナー**: クエリ/ボディ/Pub/Sub push を受理し `background_runner.run_async`（user_id=`background-worker`、session=`task-<id>`）で `background_agent` を起動。Cloud Scheduler→Pub/Sub（`{demo_id}-sched-tasks`）、`last_sched_msg_id` で再配信デデュープ、`force_run` で手動再実行。`_WORKER_SEMAPHORE` で同時実行制限。結果は `result_summary`（≤700KB、A2UI ブロック除去）として Firestore に保存。
- **トークン抽出ミドルウェア**: Authorization ヘッダ等から Bearer を抽出し `builtins._workspace_oauth_token` と session state（`GEMINI_AUTHORIZATION_ID`）に格納（Workspace MCP 用パススルー）。
- **GCS アーティファクト**: `LOGS_BUCKET_NAME` があれば `GcsArtifactService`、なければ `InMemoryArtifactService`。
- **モデル告知**: 各エージェント初回応答時に `🧠 Model: <name>` をストリームに注入。
- **インライン予算**: `INLINE_SOFT_TOOL_BUDGET_S`（既定180s）、`INLINE_HARD_DEADLINE_S`（600s）、`INLINE_IMAGE_BUDGET_S`（150s）。
- **回復力**: `LlmCallsLimitExceededError` 自動継続（最大4回）、MALFORMED リトライ（最大3回）、最終テキスト無時の無ツール再プロンプト。

### 5.5 Data Viewer

> ⚠️ エスケープ注意: Viewer は Python の `"""` 層が追加されるため、JS の `\n` には4バックスラッシュが必要（AGENTS.md §1）。

## 8. 設計上の重要な制約と安定化パッチ

**多層エスケープ（AGENTS.md §2 必読）**
- quoted ヒアドキュメント内の Python 文字列に `\n` を直書きしない → `chr(10)`
- JS `${}` と Python f-string `{}` の衝突回避 → 文字列連結を使用
- エージェント指示文に `{var}`/`{{var}}` を書かない（ADK の `inject_session_state` 正規表現 `{+[^{}]*}+` が KeyError） → `<var>` / `[VAR]`
- Code.gs 内でバッククォート3連（```）禁止 → Python 側で `chr(96)*3`
- Viewer テンプレートのみ JS `\n` は4バックスラッシュ

**スキーマ互換**: `_safe_dereference_schema` / `_ensure_types`（複雑ネスト・循環 `$ref`・`anyOf`/`oneOf` を Gemini 互換に変換、MALFORMED_FUNCTION_CALL 防止）。

**ネットワーク/セッション安定化**: HTTP/2 無効化、MCP CancelScope 修正、`ADK_ENABLE_MCP_GRACEFUL_ERROR_HANDLING=1`、300s/60s タイムアウト、非同期トークンキャッシュ。

**エージェント回復力**: ContextCache、EventsCompaction、ReflectAndRetryToolPlugin、`tool_name_prefix` による重複宣言回避、`HttpRetryOptions`（8回・2s〜60s・429/500/503）。

**デプロイ耐性**: 組織ポリシー（unauthenticated 制限）の graceful failure、静的 Agent Card、BQ 並列ロード、`--min-instances 1`、`supergateway --sessionStateless`。

**コンディショナルブロック注意（AGENTS.md §10）**: `${ flag ? \`...\` : '' }` が数百行に及ぶため、機能非依存コード（Firestore 初期化等）をフラグブロック内に誤って入れない。過去に Firestore 初期化が `enableWorkspaceMcp` ブロック内に入り、フラグ false 時に全バックグラウンドタスクが壊れた事例あり。

## 9. 既知の論点・ドキュメント乖離

調査中に判明した、既存ドキュメントと現行コードの差分:

1. **履歴/お気に入り機能の不在**: `ARCHITECTURE.md` §2.2 が記載する `getPersonalHistory` / `getGlobalHistory` / `getUsageStats` / `saveToDrive` / `restoreDemo` / `toggleFavorite` / `deleteHistoryItem` は**現行 Code.gs に存在しない**。フロントの履歴/コミュニティフィードはサーバー注入の `window._usageStats` と `localStorage` のみで構成される。Drive バックアップ・履歴復元の導線は現状無効。

2. **コード規模**: `ARCHITECTURE.md` は Code.gs を「~8,100行」、index.html を「~5,700行」とするが、現行はそれぞれ **16,498行 / 6,932行**。本体の大半（約13,600行）は `generateSetupScript` の埋め込みコード生成。

3. **必須プロパティ**: `ARCHITECTURE.md` §2.3 は `BACKUP_FOLDER_ID` を必須に挙げるが、現行 `checkConfiguration`（Code.gs:147）が検証するのは `PROJECT_ID` と `LOG_SHEET_URL` の2つのみ。

4. **三層エージェントの命名**: ルーティング指示の実装上、ワークフローパイプラインは `SCAN → CLASSIFY → PROCESS → ESCALATE → NOTIFY → REPORT`（`ARCHITECTURE.md` の `SCAN→ANALYZE→PLAN→EXECUTE→VERIFY→REPORT` とは段階名が異なる）。

> 上記乖離は、必要であれば `ARCHITECTURE.md` 側を現行コードに合わせて更新することを推奨する。
