# Generator を GAS から共有デモプロジェクト内の Cloud Run へ移行する

## 背景

現行の Generator は Google Apps Script（GAS）上の Web アプリで、中央の1デプロイを全 CE が共有する。実リソースは作らず、生成した bash セットアップスクリプトをユーザーが自分の Cloud Shell で対話実行する方式を取っている。これは「ツールが顧客プロジェクトの権限を持たなくて済む」利点がある一方、環境構築の自動化を阻む2つの壁を生んでいた——(1) 誰の権限で構築を実行するかというプロジェクト境界問題、(2) GAS の6分実行制限と shell/Docker 不可という実行基盤の制約。

## 決定

Generator を GAS から廃し、**CE/チームが所有する単一の「デモプロジェクト」内の Cloud Run アプリ**として再実装する。Generator と全 Demo を同一プロジェクトに同居させ、デモは `demo-<domain>-<suffix>` で名前空間分離する。

## 理由とトレードオフ

- **権限問題の解消**：Generator が構築先と同一プロジェクトに居るため、その SA が BigQuery / Firestore / Cloud Run をネイティブに作成できる。クロスプロジェクトの信頼や外部 SA への権限付与が不要になる。
- **実行基盤の解放**：GAS の6分制限・shell 不可がなくなり、長時間の構築を Cloud Build / Cloud Run Job に逃がせる（→ ADR-0002）。
- **失うもの**：GAS が無償提供していた Google 認証（→ Cloud Run 前段の IAP で代替）と「中央の1ツール」という配布モデル。ただしプロジェクトを *共有1個* に保つ選択により、利用統計・カタログ等の中央集権機能は1つの Firestore 上で維持される。

## 影響（Considered Options）

- 顧客ごと/デモごとに別プロジェクトを作る案は、毎回のブートストラップと中央機能の分散を招くため却下。
- 顧客の本番プロジェクトに同居させる案は、組織ポリシー・権限・ライフサイクルのリスクが最も高いため却下。
