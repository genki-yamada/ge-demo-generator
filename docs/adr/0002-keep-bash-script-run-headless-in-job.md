# bash セットアップスクリプトは温存し、ヘッドレス Job で自動実行する

## 背景

Generator を Cloud Run へ移行（ADR-0001）するにあたり、環境構築の実行方法を選び直せるようになった。同一プロジェクト化により Generator の SA が直接構築できるため、選択肢は「(A) 既存の bash スクリプト生成を維持し実行者だけ替える」「(B) provisioning を Cloud API 直接呼び出しに書き直す」「(C) Terraform 化」の3つがあった。

現行の `generateSetupScript`（約13,600行）には、Dockerfile の動的生成（条件付き MCP サイドカー部）、デプロイ順序制御、組織ポリシーの graceful failure、`--cleanup`、多数の安定化パッチが内蔵されている。

## 決定

**bash セットアップスクリプトの生成を温存**し、実行者だけを「人間の Cloud Shell」から **バックエンドがキックする Cloud Build / Cloud Run Job** に差し替える。スクリプト内の対話プロンプト（`read -p`）は全廃し、認証情報・選択値は UI で事前収集して Secret Manager 経由で Job に渡す（OAuth 同意フローは UI のブラウザで完了させ、取得トークンを Secret Manager に保存）。

## 理由とトレードオフ

- **移行リスク最小**：13,600行の実績あるロジック（Dockerfile 組み立て・デプロイ順序・組織ポリシー対応・cleanup）をそのまま活かせる。変えるのは実行トリガーのみ。
- **等価性検証が容易**：生成スクリプトを GAS 版と Node 版でバイト diff すれば移植の等価性を機械的に保証でき、ビッグバン切替前のゲートになる。
- **代償**：AGENTS.md §2 が記録する多層エスケープ地獄（`chr(10)`、バックスラッシュ4枚など）は構造的に残る。これは「文字列でコードを生成する」方式の本質的コストだが、既に飼い慣らされた既知の痛みと判断した。

## なぜ後任が疑問に思うか

「同一プロジェクトの SA で直接 Cloud API を叩けるのに、なぜわざわざ bash を生成して Job で流すのか」は自然な疑問。答えは上記の通り、API 直書きへの書き直しが Dockerfile 生成を含む大量ロジックの再実装（＝別言語での escaping バグ再発）を意味し、リスクに見合わないため。この逸脱は意図的であり「修正」してはならない。
